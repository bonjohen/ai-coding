Structured Collaborator

Structures interactions with explicit context, constraints, and verification loops.

Definition

The Structured Collaborator has moved beyond naive prompting. They provide context, specify constraints, decompose tasks, and verify output systematically.

At this level, Claude Code becomes a reliable pair programmer. The practitioner understands that prompt quality directly determines output quality and invests in framing.

Core Concepts

Context Setting

Providing relevant file context, codebase conventions, and requirements before requesting changes.

Task Decomposition

Breaking complex tasks into sequential, well-scoped prompts.

Verification Loops

Systematically testing and reviewing output after each interaction.

Constraint Specification

Explicitly stating what the output must and must not do — including edge cases, performance bounds, and style conventions.

Iterative Refinement

Using follow-up prompts to correct, extend, or improve initial output rather than accepting or rejecting wholesale.

Core Skills

Contextual Prompting

Can write prompts that include file paths, conventions, and constraints.

Observable evidence: Prompts reference specific files and coding standards.

Task Decomposition

Can break a feature request into 3-5 sequential prompts.

Observable evidence: Plans a multi-step interaction before starting.

Systematic Verification

Runs tests and reviews diffs after every significant change.

Observable evidence: Never accepts a diff without running the test suite.

Error Diagnosis

Can read Claude Code error output and provide corrective context to resolve issues.

Observable evidence: Pastes error messages back with analysis and asks for targeted fixes.

Session Management

Knows when to continue a session vs. start fresh to maintain context quality.

Observable evidence: Starts a new session when context has drifted rather than fighting accumulated confusion.

Observable Behaviors

  • Provides file paths and context at the start of each session.
  • Decomposes features into multiple focused prompts.
  • Runs tests after each change and iterates on failures.
  • Asks Claude Code to explain its reasoning when output seems wrong.

Typical Failure Modes

  • Over-specifies prompts to the point of constraining useful creativity.
  • Spends too long on prompt engineering instead of iterating.
  • Does not yet use persistent project configuration.
  • Treats each session as independent rather than building on prior work.

Evaluation Tasks

Multi-Step Feature Implementation

Implement a feature that requires changes across 3+ files using decomposed prompts.

Success Criteria

  • Feature works end-to-end
  • Each prompt is well-scoped
  • Tests pass after each step

Guided Refactoring

Refactor an existing module by providing Claude Code with constraints (preserve API, improve internal structure, add tests).

Success Criteria

  • Public API is unchanged
  • Internal structure is improved
  • New tests cover the refactored code
  • All existing tests still pass
