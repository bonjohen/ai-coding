Creating a Phased Adoption Plan for an Engineering Organization

Design a complete adoption strategy for introducing Claude Code to a 100-person engineering organization, covering training, tooling, governance, and measurement.

Scenario

You are the engineering director at a mid-size company with 100 engineers across 12 teams. The CTO has approved adopting Claude Code organization-wide. You need to create a phased plan that gets teams productive while managing risk, including a training curriculum, tooling standards, and success metrics.

Assess Current Team Capabilities

Before rolling out any tool, understand where your teams stand today. Survey engineers on their current AI coding experience and map them to the competence levels. This determines your training investment.

The assessment is not a test — it is a planning tool. Teams with more experience can move faster; teams with less experience need more support.

# adoption-plan/assessment-template.md

## AI Coding Competence Self-Assessment

For each statement, rate your current comfort level: 1 (never done) to 5 (do regularly)

### Basic Usage (Maps to Level 1)
- [ ] I can install and launch Claude Code from the command line
- [ ] I can write a prompt that produces useful code output
- [ ] I review generated diffs before accepting them
- [ ] I can recover when Claude Code produces an error or unhelpful output

### Structured Collaboration (Maps to Level 2)
- [ ] I provide file paths and context in my prompts
- [ ] I break complex tasks into multiple focused prompts
- [ ] I run tests after each AI-generated change
- [ ] I specify constraints (what the code must/must not do) in prompts

### Environment Configuration (Maps to Level 3)
- [ ] I have configured CLAUDE.md files for my projects
- [ ] I use hooks to enforce quality standards automatically
- [ ] I have set up MCP servers for external tool access
- [ ] I have configured permissions for safe AI operation

### Workflow Automation (Maps to Level 4)
- [ ] I have run multiple Claude Code agents in parallel
- [ ] I have integrated Claude Code into CI/CD pipelines
- [ ] I can design error handling for automated AI workflows
- [ ] I monitor the quality of automated AI output

Design the Training Curriculum

Structure training around the competence levels. Every engineer starts at Level 1 regardless of self-assessed skill — the training validates and reinforces fundamentals before advancing.

Each level has a hands-on workshop, self-study materials, and a practical assessment. Engineers must complete the assessment to advance. This is not bureaucracy — it is quality control.

# adoption-plan/training-curriculum.md

## Level 1: Operator (Week 1)
- **Workshop** (2 hours): Install, basic prompting, diff review, error recovery
- **Self-study**: Complete 3 guided exercises from the exercise bank
- **Assessment**: Generate a utility function, fix a bug, explain what Claude Code did
- **Graduation requirement**: Pass assessment with mentor sign-off

## Level 2: Structured Collaborator (Weeks 2-3)
- **Workshop** (3 hours): Context-setting, task decomposition, verification loops
- **Self-study**: Implement a multi-file feature using decomposed prompts
- **Assessment**: Refactor a module with constraints, all tests must pass
- **Graduation requirement**: Ship a real PR using structured collaboration

## Level 3: Environment Builder (Weeks 4-6)
- **Workshop** (3 hours): CLAUDE.md design, hooks, MCP, permissions
- **Self-study**: Configure a project environment from scratch
- **Assessment**: New team member can be productive using only your config
- **Graduation requirement**: Team project has complete Claude Code environment

## Level 4: Workflow Engineer (Weeks 7-10, select engineers)
- **Workshop** (4 hours): Multi-agent orchestration, CI/CD integration
- **Self-study**: Design and implement an automated workflow
- **Assessment**: Automated pipeline runs without manual intervention
- **Graduation requirement**: Production workflow running for 2 weeks

## Level 5: Agentic Systems Expert (Ongoing, tech leads only)
- **Seminar series** (6 sessions): Governance, safety, strategy, metrics
- **Capstone**: Design team-level governance framework
- **Ongoing**: Quarterly reviews and framework updates

Establish Tooling Standards

Standardize how Claude Code is configured across the organization. Create templates for CLAUDE.md, settings files, and CI workflows that teams can adopt and customize.

Standardization reduces support burden. When every team configures Claude Code differently, every problem is unique. When they start from a common template, solutions are reusable.

## Team-Specific

Define Success Metrics and Reporting

Define what success looks like at each phase. Metrics should cover adoption rate, productivity impact, quality maintenance, and cost. Report monthly to leadership.

Be honest about what metrics can and cannot tell you. Lines of code generated is a vanity metric. Defect rate of AI code versus human code is a real signal.

// adoption-plan/success-metrics.ts

interface AdoptionMetrics {
  // Phase 1 Success (Month 1-2): Adoption
  phase1: {
    engineersOnboarded: number;     // Target: 100% through Level 1
    engineersAtLevel2: number;      // Target: 60% by end of month 2
    projectsWithClaudeMd: number;   // Target: 80% of active repos
    weeklyActiveUsers: number;      // Target: 70% of engineers
  };

  // Phase 2 Success (Month 3-4): Productivity
  phase2: {
    prThroughputChange: number;     // Target: +20% PRs per developer
    timeToMergeChange: number;      // Target: -15% time to merge
    developerSatisfaction: number;  // Target: 4.0/5.0 on survey
    engineersAtLevel3: number;      // Target: 30% by end of month 4
  };

  // Phase 3 Success (Month 5-6): Quality & Scale
  phase3: {
    aiCodeDefectRate: number;       // Target: <= human code defect rate
    automatedWorkflowCount: number; // Target: 5+ automated workflows
    costPerDeveloperMonth: number;  // Target: < $200/developer/month
    governanceCompliance: number;   // Target: 95% of AI PRs have audit trail
  };

  // Ongoing: Health Checks
  ongoing: {
    toolAbandonment: number;        // Alert if > 10% stop using in a month
    escalationVolume: number;       // Alert if > 5 per week
    securityIncidents: number;      // Target: 0 AI-related security incidents
  };
}

Risk Mitigation and Rollback Strategy

Every adoption plan needs a rollback strategy. If AI coding is not working — if quality drops, costs spike, or teams are unhappy — you need a clear path to scale back without disruption.

The rollback strategy is not an admission of failure. It is responsible planning that makes leadership more comfortable approving the initiative.

# adoption-plan/risk-mitigation.md

## Identified Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Code quality decrease | Medium | High | Automated quality gates, mandatory tests, defect tracking |
| Cost overruns | Medium | Medium | Token budgets per team, weekly cost reports, auto-alerts at 80% |
| Security vulnerability in AI code | Low | Critical | Security scans in CI, Tier 3+ review for sensitive code |
| Developer over-reliance | Medium | Medium | Competence assessments, rotation of AI-free sprints |
| Tool vendor lock-in | Low | Medium | Abstract AI interfaces, maintain manual workflow capability |

## Rollback Triggers
- AI code defect rate exceeds 2x human code defect rate for 2 consecutive weeks
- Monthly cost exceeds budget by more than 50%
- Any security incident directly caused by AI-generated code
- Developer satisfaction drops below 3.0/5.0

## Rollback Procedure
1. **Partial rollback**: Restrict AI to Tier 1 (low-risk) code only
2. **Team-level rollback**: Specific team pauses AI usage, investigates
3. **Full rollback**: All automated AI workflows paused, manual review only
4. **Post-mortem**: Root cause analysis within 1 week, remediation plan

All rollback decisions require sign-off from Engineering Director.
All rollbacks are temporary — the goal is to fix and resume, not to abandon.
