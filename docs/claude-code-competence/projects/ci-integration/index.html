Adding Claude Code to a CI/CD Pipeline for Automated PR Review

Integrate Claude Code into a GitHub Actions workflow to automatically review pull requests and post feedback.

Scenario

Your team merges 15-20 PRs per week. Manual code review is a bottleneck. You want Claude Code to automatically review every PR, check for common issues, and post comments — not to replace human review, but to catch the easy things so reviewers can focus on design and architecture.

Design the Review Workflow

The CI workflow should: (1) trigger on PR open and update, (2) run Claude Code in headless mode to review the diff, (3) post review comments back to the PR, and (4) set a status check that does not block merge but provides visibility.

This is a quality monitoring system. It augments human review rather than replacing it.

Create the GitHub Actions Workflow

The workflow uses Claude Code in headless mode with a structured prompt that asks for specific review criteria. The output is captured and posted as a PR comment.

Store the API key as a GitHub secret. Never hardcode credentials in workflow files.

# .github/workflows/claude-review.yml
name: Claude Code PR Review

on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Get PR diff
        run: git diff origin/$...HEAD > /tmp/pr-diff.txt

      - name: Run Claude Code Review
        env:
          ANTHROPIC_API_KEY: $
        run: |
          claude -p "Review the following code diff for a pull request. \
          Check for: (1) bugs or logic errors, (2) security issues, \
          (3) missing error handling, (4) test coverage gaps, \
          (5) naming and convention violations per CLAUDE.md. \
          Format your response as a markdown review with sections for \
          each category. Only mention issues you actually found — do not \
          list categories with no issues. Be specific: reference file \
          names and line numbers. Here is the diff:" \
          --file /tmp/pr-diff.txt > /tmp/review-output.md

      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('/tmp/review-output.md', 'utf8');
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## Claude Code Review\n\n${review}\n\n---\n*Automated review by Claude Code. This does not replace human review.*`
            });

Add Cost Controls and Rate Limiting

Automated pipelines can run up costs quickly. Add controls to limit token usage per review and skip reviews on trivial changes.

Set a maximum output token limit and filter out PRs that only change documentation or configuration.

# Add to the workflow, before the review step:
      - name: Check if review needed
        id: check
        run: |
          # Skip review for docs-only or config-only changes
          CHANGED_FILES=$(git diff --name-only origin/$...HEAD)
          CODE_FILES=$(echo "$CHANGED_FILES" | grep -E '\.(ts|js|tsx|jsx|py|go|rs)$' || true)
          if [ -z "$CODE_FILES" ]; then
            echo "skip=true" >> $GITHUB_OUTPUT
            echo "No code files changed, skipping review."
          else
            echo "skip=false" >> $GITHUB_OUTPUT
            # Limit diff size to control token usage
            DIFF_LINES=$(wc -l < /tmp/pr-diff.txt)
            if [ "$DIFF_LINES" -gt 2000 ]; then
              echo "large_diff=true" >> $GITHUB_OUTPUT
            else
              echo "large_diff=false" >> $GITHUB_OUTPUT
            fi
          fi

      - name: Run Claude Code Review
        if: steps.check.outputs.skip != 'true'
        env:
          ANTHROPIC_API_KEY: $
        run: |
          MAX_TOKENS=4096
          if [ "$" = "true" ]; then
            echo "Large diff detected. Reviewing summary only."
            PROMPT="Summarize the key changes and flag any obvious issues."
          else
            PROMPT="Full review as specified in the review prompt."
          fi
          claude -p "$PROMPT" --max-tokens $MAX_TOKENS \
            --file /tmp/pr-diff.txt > /tmp/review-output.md

Monitor Review Quality Over Time

Track whether the automated reviews are actually useful. Log review outputs and periodically check: are the comments actionable? Are there too many false positives? Is the team ignoring the automated reviews?

If the team starts ignoring automated reviews, the system is adding noise, not value. Adjust the prompt to be more precise or raise the threshold for what gets flagged.

# Add a simple tracking step to the workflow
      - name: Log Review Metrics
        if: steps.check.outputs.skip != 'true'
        run: |
          # Count issues found by category
          BUGS=$(grep -c '## Bugs' /tmp/review-output.md || echo 0)
          SECURITY=$(grep -c '## Security' /tmp/review-output.md || echo 0)
          echo "review_date=$(date -u +%Y-%m-%d)" >> review-metrics.log
          echo "pr_number=$" >> review-metrics.log
          echo "diff_lines=$(wc -l < /tmp/pr-diff.txt)" >> review-metrics.log
          echo "issues_found=$(grep -c '^-' /tmp/review-output.md || echo 0)" >> review-metrics.log
