Designing an AI Coding Governance Policy

Create a comprehensive governance framework for AI-generated code covering review processes, audit trails, risk tiers, and safety boundaries.

Scenario

Your engineering organization of 50 developers is adopting Claude Code across all teams. Leadership needs a governance framework that ensures AI-generated code meets quality and security standards without slowing down development. You need to design the policy, tooling, and processes.

Define Risk Tiers for AI-Generated Code

Not all code carries the same risk. Internal tooling has different stakes than payment processing. Define tiers that determine how much oversight AI-generated code requires.

The tiers should map to concrete review requirements. Higher-risk code gets more human review; lower-risk code can rely more heavily on automated checks.

# ai-governance/risk-tiers.md

## Risk Tier Definitions

### Tier 1 — Low Risk
**Scope**: Internal tools, documentation, test code, prototypes
**AI Review Policy**: Automated checks sufficient. Human review optional.
**Examples**: Admin dashboards, dev scripts, test utilities, markdown docs
**Required Checks**: Linting, type-checking, automated test pass

### Tier 2 — Medium Risk
**Scope**: Non-critical application features, internal APIs, batch jobs
**AI Review Policy**: Automated checks + one human reviewer
**Examples**: Reporting features, notification systems, data exports
**Required Checks**: All Tier 1 + security scan + one human approval

### Tier 3 — High Risk
**Scope**: User-facing features, authentication, data handling, external APIs
**AI Review Policy**: Automated checks + two human reviewers (one must be senior)
**Examples**: Auth flows, payment integration, PII handling, public APIs
**Required Checks**: All Tier 2 + senior review + security team sign-off

### Tier 4 — Critical
**Scope**: Infrastructure, deployment, security controls, compliance code
**AI Review Policy**: AI assistance allowed for drafting, but final code must be
human-written or line-by-line human-verified with sign-off
**Examples**: CI/CD pipelines, IAM policies, encryption, audit logging
**Required Checks**: All Tier 3 + compliance review + documented justification

Design the Audit Trail System

Every piece of AI-generated code should be traceable. Design an audit system that records what was generated, by which model, with what prompt, and who reviewed it.

The audit trail serves two purposes: accountability (who approved this code?) and learning (what prompts produce good results?).

// ai-governance/audit-schema.ts

// Every AI-assisted commit is tagged with metadata
interface AICodeAuditRecord {
  // Identification
  commitHash: string;
  pullRequestId: number;
  timestamp: string;

  // AI Generation Details
  model: string;             // e.g., "claude-opus-4-6"
  toolVersion: string;       // e.g., "claude-code@1.2.3"
  promptSummary: string;     // Summary of the prompt (not full prompt to avoid IP issues)
  filesModified: string[];   // List of files touched by AI
  linesGenerated: number;    // Lines of code generated by AI
  linesModified: number;     // Lines of existing code modified by AI

  // Review Details
  riskTier: 1 | 2 | 3 | 4;
  reviewers: string[];       // GitHub usernames of human reviewers
  automatedChecks: {
    name: string;            // e.g., "eslint", "type-check", "security-scan"
    passed: boolean;
    details?: string;
  }[];

  // Approval
  approved: boolean;
  approvedBy: string;
  approvalNotes?: string;
}

Define Safety Boundaries for Autonomous Agents

When Claude Code runs in CI/CD or in autonomous workflows, it needs hard boundaries. Define what agents are allowed to do and what requires human escalation.

Safety boundaries should be enforced technically (through permissions and hooks), not just through policy documents that people might not read.

# ai-governance/safety-boundaries.md

## Autonomous Agent Boundaries

### Allowed Without Human Approval
- Read any file in the repository
- Run test suites and linting
- Generate code in feature branches
- Post review comments on pull requests
- Create draft pull requests

### Requires Human Approval Before Execution
- Merge any pull request
- Modify CI/CD pipeline configuration
- Change dependency versions
- Modify environment variables or secrets references
- Delete files or branches
- Access production systems or data

### Never Allowed (Hard Block)
- Push directly to main/production branches
- Access or modify production databases
- Modify security controls or IAM policies
- Publish packages to registries
- Create or modify cloud infrastructure
- Access secrets or credentials directly

## Escalation Protocol
1. Agent encounters a blocked operation
2. Agent creates a GitHub issue tagged `ai-escalation`
3. On-call engineer reviews within 4 hours
4. Engineer either approves (with audit note) or rejects with guidance

Design Metrics for Monitoring AI Coding Adoption

You cannot govern what you cannot measure. Define metrics that track both the productivity benefits and the risks of AI coding adoption.

The metrics should answer: Is AI coding making us faster? Is it maintaining quality? Are we catching issues before they reach production?

// ai-governance/metrics.ts

// Productivity Metrics
interface ProductivityMetrics {
  prThroughput: number;           // PRs merged per developer per week
  timeToMerge: number;            // Average hours from PR open to merge
  aiAssistedPRRatio: number;      // % of PRs with AI-generated code
  linesGeneratedPerWeek: number;  // Total AI-generated lines across org
}

// Quality Metrics
interface QualityMetrics {
  aiCodeDefectRate: number;       // Bugs per 1000 lines of AI code
  humanCodeDefectRate: number;    // Bugs per 1000 lines of human code
  reviewRejectionRate: number;    // % of AI PRs requiring revision
  automatedCheckFailRate: number; // % of AI commits failing CI
}

// Safety Metrics
interface SafetyMetrics {
  escalationCount: number;        // AI escalations per week
  boundaryViolationAttempts: number; // Blocked operations per week
  auditCompleteness: number;      // % of AI commits with full audit records
  meanTimeToEscalationResponse: number; // Hours from escalation to response
}

// Cost Metrics
interface CostMetrics {
  tokenUsagePerWeek: number;      // Total tokens consumed
  costPerPR: number;              // Average API cost per PR review
  costPerDeveloper: number;       // Monthly cost per developer
  roiEstimate: number;            // Estimated time saved * developer cost
}

Create the Rollout and Exception Policy

Governance only works if it has a clear rollout plan and a process for exceptions. Teams that find the policy too restrictive need a path to request adjustments, not a reason to circumvent the system.

Build in a feedback loop: review the governance framework quarterly and adjust based on metrics and team feedback.

# ai-governance/rollout.md

## Rollout Plan

### Phase 1 (Weeks 1-4): Pilot
- 2 volunteer teams adopt the framework
- All AI code is Tier 3 review (extra scrutiny during pilot)
- Weekly feedback sessions with pilot teams
- Metrics collection begins

### Phase 2 (Weeks 5-8): Controlled Expansion
- Expand to 5 teams based on pilot learnings
- Risk tiering goes into effect (not all code needs Tier 3)
- Automated audit trail tooling deployed
- First metrics review

### Phase 3 (Weeks 9-12): General Availability
- All teams can adopt
- Self-service onboarding with team-specific CLAUDE.md templates
- Full metrics dashboard available
- Quarterly governance review established

## Exception Process
1. Team submits exception request via GitHub issue template
2. Request includes: what rule, why exception, proposed alternative safeguard
3. Security team reviews within 1 week
4. Approved exceptions are time-boxed (max 90 days) and logged
5. Permanent policy changes go through quarterly review
