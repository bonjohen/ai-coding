Implementing a Feature Across Multiple Files

Decompose a feature request into focused prompts and implement it across model, service, and API layers.

Scenario

You need to add a 'tags' feature to a blog application. Posts should support multiple tags, tags should be searchable, and the API needs new endpoints. This touches the database model, service layer, and route handlers.

Plan the Task Decomposition

Before writing any prompts, plan the sequence of changes. A feature that spans multiple files should be broken into steps that each leave the codebase in a working state.

For this feature, the natural order is: (1) add the data model, (2) update the service layer, (3) add API routes, (4) write tests for each layer. Each step is one focused prompt.

Step 1: Add the Tag Model

Start with the data layer. Provide Claude Code with context about the existing model structure and ask it to add the Tag model and the relationship to Post.

Notice how the prompt references specific files and existing patterns. This is what separates Level 2 from Level 1 — you are setting context, not just asking for code.

> Look at the existing Post model in src/models/post.ts and the
> database schema in prisma/schema.prisma. Add a Tag model with
> fields: id (auto-increment), name (unique string, max 50 chars),
> and slug (unique string). Create a many-to-many relationship
> between Post and Tag. Follow the same naming conventions used
> by the existing models. Generate a Prisma migration.

Step 2: Update the Service Layer

After verifying the model changes compile and the migration runs, move to the service layer. Reference the patterns established in the existing service files.

Run tests after this step before moving on. Each step should leave the codebase in a passing state.

> Now look at src/services/post.service.ts for the existing service
> patterns. Create src/services/tag.service.ts with these operations:
> - createTag(name: string): creates a tag, auto-generates slug
> - getTagBySlug(slug: string): returns tag or null
> - addTagsToPost(postId: number, tagIds: number[]): links tags to post
> - getPostsByTag(tagSlug: string, page: number, limit: number): paginated
>
> Follow the same error handling pattern used in post.service.ts.
> Use the Prisma client singleton from src/lib/prisma.ts.

Step 3: Add API Routes

With the service layer in place and tested, add the API routes. Specify constraints about authentication and validation.

> Look at src/routes/post.routes.ts for the existing route patterns.
> Create src/routes/tag.routes.ts with:
> - GET /tags — list all tags (public)
> - GET /tags/:slug/posts — get posts by tag, paginated (public)
> - POST /posts/:id/tags — add tags to a post (requires auth)
> - DELETE /posts/:id/tags/:tagId — remove tag from post (requires auth)
>
> Use the same validation middleware pattern from post.routes.ts.
> Auth routes should use the requireAuth middleware from src/middleware/auth.ts.
> Register the new routes in src/routes/index.ts.

Step 4: Run Tests and Verify

After each step, you should have been running tests. Now do a final end-to-end verification. Ask Claude Code to write integration tests if they do not already exist.

The verification loop is what makes Level 2 reliable. You never move to the next step until the current step is verified.

> Write integration tests for the tag feature in
> tests/integration/tags.test.ts. Test:
> - Creating a tag
> - Adding tags to a post
> - Fetching posts by tag with pagination
> - Auth is required for write operations
> - Duplicate tag names are rejected
>
> Follow the test patterns in tests/integration/posts.test.ts.
> Use the same test database setup and teardown helpers.
