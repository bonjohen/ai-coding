Refactoring a Module with Constraints

Guide Claude Code through a refactoring that preserves the public API while improving internal structure and adding tests.

Scenario

Your authentication module has grown into a 500-line monolith. You need to refactor it into smaller, focused files while keeping the exact same public API so that no consuming code needs to change.

Establish the Constraints

The most important part of a constrained refactoring is making the constraints explicit. Tell Claude Code exactly what must not change, what must be preserved, and what the goals are.

This is constraint specification — a core Level 2 skill. Vague instructions like 'refactor this' will produce unpredictable results.

> I need to refactor src/auth/index.ts (currently ~500 lines) into
> smaller modules. Here are the hard constraints:
>
> 1. The public API exported from src/auth/index.ts MUST NOT change.
>    Any file importing from 'src/auth' must work without modification.
> 2. All existing tests in tests/auth.test.ts must continue to pass.
> 3. No new dependencies may be added.
>
> Goals:
> - Split into: token.ts (JWT logic), password.ts (hashing/comparison),
>   session.ts (session management), middleware.ts (Express middleware)
> - Each sub-module should have a single responsibility
> - Re-export everything from index.ts to preserve the public API
>
> First, show me the plan before making any changes.

Review the Refactoring Plan

Claude Code should present a plan before writing code. Review it to make sure the proposed file structure makes sense and the re-export strategy will preserve the public API.

If anything in the plan concerns you, now is the time to correct it. It is much cheaper to fix a plan than to fix code.

Execute the Refactoring Incrementally

Ask Claude Code to do the refactoring one file at a time. After each file is extracted, run the test suite. This is iterative refinement in practice — each step is verified before moving on.

If a test breaks after extracting a module, do not move on. Diagnose the issue with Claude Code and fix it first.

> Let's start with extracting the JWT logic. Move the token generation
> and verification functions into src/auth/token.ts. Update index.ts
> to import and re-export from token.ts. Do not change any function
> signatures.

# After Claude Code makes the changes:
npm test -- tests/auth.test.ts

# If tests pass, continue to the next extraction:
> Good, tests pass. Now extract the password hashing logic into
> src/auth/password.ts. Same rules: preserve function signatures,
> re-export from index.ts.

Add Unit Tests for New Modules

Once the refactoring is complete and all existing tests pass, ask Claude Code to add unit tests for each new sub-module. The existing tests are integration-level; now each module deserves its own focused tests.

> All existing tests pass after the refactoring. Now add unit tests:
> - tests/auth/token.test.ts — test JWT generation, verification,
>   expiration, and invalid token handling
> - tests/auth/password.test.ts — test hashing, comparison, and
>   edge cases (empty string, very long input)
>
> Follow the test style in the existing tests/auth.test.ts.
> Each test file should be self-contained with its own setup.
