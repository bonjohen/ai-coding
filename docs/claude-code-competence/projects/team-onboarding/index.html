Configuring an Environment for Team Onboarding

Set up a project environment so a new team member can be productive with Claude Code immediately without verbal instructions.

Scenario

A new engineer is joining your team next week. They have used Claude Code before but know nothing about your project. You want them to be able to clone the repo, launch Claude Code, and start contributing on day one without anyone walking them through the setup.

Write an Onboarding-Focused CLAUDE.md

Your CLAUDE.md needs to serve double duty: it guides Claude Code, and it guides the human reading it. A new team member should be able to read the CLAUDE.md and understand how to work in this project.

Include setup instructions, common workflows, and gotchas. Think of it as a living README that also happens to configure Claude Code.

# CLAUDE.md

## Getting Started (New Team Members)
1. Copy `.env.example` to `.env` and fill in values from 1Password vault "Engineering"
2. Run `docker compose up -d` to start PostgreSQL and Redis
3. Run `npm install && npm run db:migrate && npm run db:seed`
4. Run `npm test` to verify everything works

## Common Workflows
- **Add a new API endpoint**: Create route in src/routes/, service in src/services/, test in tests/
- **Add a database migration**: Edit prisma/schema.prisma, run `npx prisma migrate dev --name description`
- **Debug a failing test**: Run `npm test -- --verbose tests/path/to/test.ts`

## Architecture
- Monorepo: packages/api (Express), packages/shared (types + utils), packages/worker (background jobs)
- All cross-package imports go through packages/shared
- API follows REST conventions; see src/routes/users.ts as the canonical example

## Conventions
- Branch naming: feature/TICKET-description, fix/TICKET-description
- Commit messages: conventional commits (feat:, fix:, chore:, etc.)
- All PRs require tests. No exceptions.
- Use zod for request validation, never manual type checks

## Things That Will Bite You
- The test database is wiped between test files but NOT between tests in the same file. Use beforeEach for cleanup.
- Redis must be running for the session tests. If sessions tests fail, check `docker compose ps`.
- The worker package has its own tsconfig. Don't reference api types from worker directly.

Configure Permissions for Safe Exploration

A new team member will be exploring the codebase. Configure permissions so Claude Code can read anything but is restricted in what it writes and executes. This protects against accidental damage while still being productive.

Think of permissions as training wheels — they prevent costly mistakes while the new person learns the project.

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Read(*)",
      "Bash(npm run *)",
      "Bash(npm test *)",
      "Bash(npx prisma *)",
      "Bash(docker compose ps)",
      "Bash(docker compose logs *)",
      "Write(src/**)",
      "Write(tests/**)",
      "Write(packages/*/src/**)"
    ],
    "deny": [
      "Write(.env*)",
      "Write(docker-compose.yml)",
      "Write(prisma/migrations/**)",
      "Bash(docker compose down *)",
      "Bash(npm publish *)",
      "Bash(git push --force *)",
      "Bash(rm -rf *)"
    ]
  }
}

Set Up Layered Context for Each Package

In a monorepo, each package has its own conventions. Create CLAUDE.md files in each package directory so Claude Code gets the right context regardless of where the team member is working.

# packages/api/CLAUDE.md

## API Package
Express REST API. Entry point: src/index.ts.

## Route Handler Pattern
Route handlers should:
1. Validate input with zod schema
2. Call service method
3. Return result with appropriate HTTP status
4. Never contain business logic directly

See src/routes/users.ts as the reference implementation.

---

# packages/worker/CLAUDE.md

## Worker Package
BullMQ background job processors. Each job type has its own file in src/jobs/.

## Job Pattern
- Export a processor function and a queue configuration
- Jobs must be idempotent (safe to retry)
- Always log job start, completion, and failure
- Use packages/shared for any types shared with the API

Test the Onboarding Experience

Before the new team member arrives, test the onboarding yourself. Clone the repo into a fresh directory, launch Claude Code, and try to complete a typical task using only the CLAUDE.md files for guidance.

If you find yourself needing to explain something verbally, that information should go into a CLAUDE.md file.

# Clone into a fresh directory to simulate the new member's experience
git clone git@github.com:your-org/your-repo.git /tmp/fresh-clone
cd /tmp/fresh-clone

# Launch Claude Code and try a typical onboarding task
claude

> I'm new to this project. Help me get set up and then implement
> a simple feature: add a GET /api/health endpoint that returns
> the current time and database connection status.
