Setting Up CLAUDE.md, Hooks, and MCP for a New Project

Configure a complete Claude Code environment with layered context files, automated hooks, and an MCP server integration.

Scenario

You are starting a new TypeScript microservice and want to configure the project so that Claude Code is effective from the first session. This means setting up CLAUDE.md with project conventions, hooks for quality enforcement, and an MCP server for database access.

Design the Root CLAUDE.md

The root CLAUDE.md is the most important configuration file. It tells Claude Code about the project's architecture, conventions, build commands, and rules. Keep it focused — include what Claude Code needs to know, not everything about the project.

A good CLAUDE.md answers three questions: How do I build and test this project? What conventions should I follow? What should I never do?

# CLAUDE.md

## Project Overview
Order processing microservice. TypeScript, Express, Prisma ORM, PostgreSQL.

## Build & Test Commands
- `npm run build` — compile TypeScript
- `npm test` — run all tests (Jest)
- `npm test -- tests/unit/FILE.test.ts` — run a single test file
- `npm run lint` — ESLint + Prettier check
- `npm run db:migrate` — run Prisma migrations

## Code Conventions
- Use named exports, never default exports
- All service functions return `Result` (see src/lib/result.ts)
- Database access only through Prisma — never raw SQL
- Error messages must be user-facing safe (no stack traces, no internal IDs)

## Architecture
- src/routes/ — Express route handlers (thin, delegate to services)
- src/services/ — Business logic (no direct HTTP concerns)
- src/models/ — Prisma schema and generated types
- src/lib/ — Shared utilities

## Rules
- NEVER modify prisma/migrations/ manually
- NEVER commit .env files
- Always run `npm test` before considering a task complete

Add Directory-Level CLAUDE.md Files

For larger projects, add CLAUDE.md files in key directories to provide scoped context. These supplement the root file with directory-specific conventions.

Directory-level files should be short and focused. They answer: what is in this directory, and what rules apply here specifically?

# src/services/CLAUDE.md

## Service Layer Conventions
- Every service file exports a class with static methods
- Methods accept plain objects, not Express Request objects
- All methods must validate input using zod schemas from src/schemas/
- Return Result — never throw exceptions
- Log operations using the logger from src/lib/logger.ts

## Testing
- Unit tests mock the Prisma client using tests/helpers/mock-prisma.ts
- Each service file has a corresponding test in tests/unit/services/

Configure Hooks for Quality Enforcement

Hooks run automatically during Claude Code's workflow to catch issues before they are committed. Configure hooks in the project settings to enforce your quality standards.

The key insight at Level 3 is that hooks shift enforcement from your attention to automation. You do not have to remember to run the linter — the hook does it for you.

// .claude/settings.json
{
  "hooks": {
    "preCommit": [
      {
        "command": "npm run lint",
        "description": "Run ESLint and Prettier checks"
      },
      {
        "command": "npm test -- --bail",
        "description": "Run test suite, stop on first failure"
      }
    ],
    "postFileEdit": [
      {
        "command": "npx tsc --noEmit --pretty",
        "description": "Type-check after file edits"
      }
    ]
  },
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(npx prisma *)",
      "Bash(npx tsc *)",
      "Read(*)",
      "Write(src/**)",
      "Write(tests/**)"
    ],
    "deny": [
      "Write(.env*)",
      "Write(prisma/migrations/**)",
      "Bash(rm -rf *)",
      "Bash(git push *)"
    ]
  }
}

Set Up an MCP Server for Database Access

MCP (Model Context Protocol) servers give Claude Code access to external tools and data sources. For this project, set up a PostgreSQL MCP server so Claude Code can query the development database to understand the current state of data.

This is powerful but requires care. The MCP server should connect to the development database only, never production.

// .claude/mcp.json
{
  "servers": {
    "postgres-dev": {
      "command": "npx",
      "args": [
        "@anthropic/mcp-server-postgres",
        "--connection-string",
        "postgresql://dev:dev@localhost:5432/orders_dev"
      ],
      "description": "Development database access for querying schema and sample data"
    }
  }
}

Verify the Complete Setup

Launch a new Claude Code session and verify that everything works together. The CLAUDE.md should be picked up automatically, hooks should run on edits, and the MCP server should be available.

Ask Claude Code to describe the project to confirm it has read the CLAUDE.md. Then make a small change to trigger the hooks and verify they fire.

claude

> Describe this project's architecture and conventions based on
> the CLAUDE.md files.

# Claude Code should accurately describe the project from CLAUDE.md

> Now create a simple health check endpoint in src/routes/health.ts
> that returns { status: 'ok', timestamp: Date.now() }. Register it
> in src/routes/index.ts.

# The hooks should run automatically: type-check on file edit,
# lint + test on commit attempt
