Running Multiple Claude Code Agents on Parallel Worktrees

Orchestrate multiple Claude Code instances working on independent tasks using git worktrees for isolation.

Scenario

You have a sprint with four independent tasks: a new API endpoint, a database migration, a test suite expansion, and a documentation update. Instead of doing them sequentially, you want to run four Claude Code agents in parallel, each on its own worktree, and merge the results.

Design the Task Dependency Graph

Before launching agents, analyze the tasks for dependencies. Tasks that are truly independent can run in parallel. Tasks with dependencies must be sequenced.

In this case: the API endpoint and docs are independent, the test suite expansion depends on no new code, and the migration is independent. All four can run in parallel.

# Task dependency analysis:
# Task A: New /api/reports endpoint     — independent
# Task B: Add indexes to orders table   — independent
# Task C: Expand test coverage for auth  — independent (tests existing code)
# Task D: Update API documentation       — independent
#
# Result: All four tasks can run in parallel.
# Merge order: B (migration) first, then A, C, D in any order.

Create Git Worktrees for Isolation

Each agent needs its own copy of the repository to avoid file conflicts. Git worktrees are lightweight — they share the .git directory but have independent working directories.

Create a worktree for each task on its own branch.

# Create worktrees for each parallel task
git worktree add ../project-agent-a -b feature/reports-endpoint
git worktree add ../project-agent-b -b feature/order-indexes
git worktree add ../project-agent-c -b feature/auth-test-coverage
git worktree add ../project-agent-d -b feature/api-docs-update

# Verify worktrees
git worktree list

Launch Agents in Headless Mode

Launch each agent in non-interactive (headless) mode with a clear prompt. The -p flag sends a prompt and exits when done. Each agent runs in its own worktree directory.

Use detailed prompts since there is no opportunity for interactive follow-up in headless mode. Include all context and constraints upfront.

# Launch all four agents in parallel using headless mode
# Agent A: New API endpoint
cd ../project-agent-a && claude -p "Create a new GET /api/reports endpoint \
in src/routes/reports.ts that returns order statistics: total orders, \
revenue by month, and top products. Use the existing service patterns \
from src/services/orders.service.ts. Add tests in \
tests/integration/reports.test.ts. Run npm test before finishing." &

# Agent B: Database migration
cd ../project-agent-b && claude -p "Add database indexes to improve query \
performance on the orders table. Add indexes on: customer_id, created_at, \
status, and a composite index on (customer_id, created_at). Create a \
Prisma migration. Run the migration against the dev database and verify \
with npm test." &

# Agent C: Test coverage expansion
cd ../project-agent-c && claude -p "Expand test coverage for the auth \
module. Current coverage is at 60%. Add tests for: token refresh edge \
cases, session expiration, concurrent login handling, and password reset \
flow. Target 90% coverage. Check coverage with npm test -- --coverage." &

# Agent D: Documentation update
cd ../project-agent-d && claude -p "Update the API documentation in \
docs/api.md to include all endpoints in src/routes/. For each endpoint, \
document: method, path, request body schema, response schema, auth \
requirements, and example curl command. Follow the existing format in \
the file." &

# Wait for all agents to complete
wait
echo "All agents finished."

Review Results and Merge

After all agents finish, review each branch's changes before merging. Check for quality, correctness, and any unexpected modifications.

Merge in dependency order: migration first (since other code might depend on schema), then the rest.

# Review each branch
cd ../project-agent-b && git diff main...HEAD --stat
cd ../project-agent-a && git diff main...HEAD --stat
cd ../project-agent-c && git diff main...HEAD --stat
cd ../project-agent-d && git diff main...HEAD --stat

# Merge in order (migration first)
cd /path/to/main/repo
git merge feature/order-indexes --no-ff -m "feat: add order table indexes"
git merge feature/reports-endpoint --no-ff -m "feat: add reports API endpoint"
git merge feature/auth-test-coverage --no-ff -m "test: expand auth test coverage"
git merge feature/api-docs-update --no-ff -m "docs: update API documentation"

# Run full test suite after all merges
npm test

# Clean up worktrees
git worktree remove ../project-agent-a
git worktree remove ../project-agent-b
git worktree remove ../project-agent-c
git worktree remove ../project-agent-d

Handle Merge Conflicts

When parallel agents modify related files, merge conflicts are expected. Use Claude Code to help resolve them intelligently.

The key insight is that conflict resolution is itself a task you can delegate to Claude Code, as long as you verify the result.

# If a merge conflict occurs:
git merge feature/reports-endpoint --no-ff
# CONFLICT in src/routes/index.ts

# Use Claude Code to resolve the conflict
claude -p "There is a merge conflict in src/routes/index.ts from merging \
the reports-endpoint branch. Resolve the conflict by keeping both the \
existing routes and the new reports routes. Both sets of routes should be \
registered. Show me the resolved file."
