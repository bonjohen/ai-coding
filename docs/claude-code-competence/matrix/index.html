Competency Matrix

Full competency matrix mapping dimensions against the five levels.

Dimension L1: Operator L2: Collaborator L3: Builder L4: Engineer L5: Expert
Conceptual Understanding
Knows Claude Code exists and can run it.
  • Understands it is a CLI tool that generates code from natural language
  • Cannot explain what a context window is or why it matters
  • Does not distinguish between tool use and direct text generation
  • Treats the model as a black box — no mental model of how it works
  • Cannot predict when the tool will struggle or succeed
Understands the prompt-response cycle and role of context.
  • Knows context window limits exist and affect multi-file tasks
  • Understands that providing file paths and conventions improves output
  • Can explain the difference between a vague and a well-structured prompt
  • Recognizes when output quality degrades due to insufficient context
  • Begins to understand that the same prompt can produce different results
Understands configuration, hooks, MCP, and persistent context.
  • Knows how CLAUDE.md content shapes model behavior across sessions
  • Understands MCP server architecture and what data they expose
  • Can explain how hooks intercept and automate tool events
  • Understands the permission model and why allowlists/denylists matter
  • Grasps the difference between session context and persistent configuration
Understands multi-agent orchestration, automation, and system design.
  • Grasps headless mode and non-interactive operation constraints
  • Understands how worktree isolation prevents agent conflicts
  • Can reason about token costs and budget across parallel workflows
  • Understands the failure modes of automated agent pipelines
  • Can articulate trade-offs between speed, cost, and quality in agent workflows
Deep architectural understanding of AI coding systems at organizational scale.
  • Can reason about system-level implications of widespread AI code generation
  • Understands how AI coding governance parallels data governance maturity models
  • Can evaluate model capabilities against organizational risk profiles
  • Understands the human factors of AI-assisted development at scale
  • Can predict how tool evolution will affect organizational strategy
Task Framing
Vague, single-turn prompts with no structure.
  • Asks for things without specifying constraints or edge cases
  • Does not decompose multi-step tasks into sequential prompts
  • Provides no file paths, conventions, or architectural context
  • Expects the model to guess what 'fix this' or 'make it work' means
  • Retries the same prompt when results are poor instead of refining
Structured prompts with explicit context, constraints, and decomposition.
  • Provides file paths, function names, and coding conventions in prompts
  • Decomposes features into 3-5 sequential, focused prompts
  • Specifies constraints: edge cases, performance bounds, style rules
  • References previous steps when building on prior work in a session
  • Plans the prompt sequence before starting a complex task
Environment-level framing that reduces per-prompt overhead.
  • Context is provided persistently via CLAUDE.md, not repeated each prompt
  • Tasks build on established project configuration and conventions
  • Prompts assume the model already knows build commands and architecture
  • Frames tasks in terms of project patterns rather than low-level instructions
  • New sessions are productive immediately because environment provides context
Workflow-level task design across multiple agents and pipelines.
  • Decomposes work across multiple agents with clear interface contracts
  • Designs task dependency graphs identifying what can run in parallel
  • Frames headless/scripted prompts for batch and CI/CD operation
  • Specifies rollback and error-handling behavior in automated task definitions
  • Writes prompt templates that are reusable across similar tasks
Organizational task strategy and scalable prompt patterns.
  • Frames AI coding work at portfolio level across teams and projects
  • Designs task patterns and prompt libraries that scale across the organization
  • Creates standards for how teams decompose work for AI agents
  • Evaluates task framing effectiveness using quality and productivity metrics
  • Adapts task strategies as model capabilities evolve
Verification Discipline
Minimal — accepts output if it looks roughly correct.
  • Does not run tests after accepting AI-generated code
  • Reviews output superficially — scans rather than reads line-by-line
  • Cannot identify subtle bugs in generated diffs
  • Accepts hardcoded values, missing error handling, or style violations
  • Does not verify that generated code handles edge cases
Systematic — reviews diffs and tests after every change.
  • Reviews every diff line-by-line before accepting
  • Runs the full test suite after each significant change
  • Checks for hardcoded values that should come from configuration
  • Verifies naming conventions and code style match project standards
  • Iterates on failures rather than accepting partially correct output
Automated — hooks and tooling enforce verification standards.
  • Pre-commit hooks run linting, type-checking, and formatting automatically
  • Configured hooks catch common AI-generated code issues before commit
  • Security scanning runs on every change without manual invocation
  • Test coverage thresholds prevent merging untested AI-generated code
  • Verification is part of the environment, not dependent on individual discipline
Pipeline-integrated verification with automated quality gates.
  • CI/CD runs full verification on every agent-generated commit
  • Automated quality gates block merges that fail verification thresholds
  • Agent output quality is monitored for drift over time
  • Verification results are logged and tracked as metrics
  • Failure patterns in automated verification trigger process improvements
Governance-level verification with audit trails and compliance.
  • Audit trails record every AI-generated change with provenance
  • Compliance verification is integrated into deployment workflows
  • Verification policies are documented and enforced across teams
  • Regular audits assess whether verification processes catch real issues
  • Incident post-mortems track whether AI-generated code was the root cause
Context Management
No context management — prompts without orientation.
  • Starts prompting without telling the model what project or file is in scope
  • Loses context in long sessions without realizing quality has degraded
  • Does not know when to start a fresh session vs. continue
  • Provides the same context repeatedly because it is not persisted
  • Cannot explain why the model sometimes 'forgets' earlier instructions
Per-session context management with deliberate orientation.
  • Provides file paths and architecture context at the start of each session
  • Recognizes when context window is approaching limits and starts fresh
  • References specific files and functions rather than vague descriptions
  • Knows that long conversations accumulate noise and reduces quality
  • Begins saving useful context snippets for reuse across sessions
Persistent project context via CLAUDE.md and configuration.
  • Root CLAUDE.md provides baseline project context automatically
  • Directory-level CLAUDE.md files scope context to specific code areas
  • Permission settings reduce context overhead from repeated approval prompts
  • MCP servers provide live context from databases and external systems
  • Context hierarchy is designed so the model gets the right info at the right scope
Cross-agent context coordination across parallel workflows.
  • Context is explicitly partitioned when work is split across agents
  • Shared interfaces and type definitions serve as context contracts between agents
  • Worktree isolation ensures agents have clean context without interference
  • Context budgets are managed to control token costs in multi-agent workflows
  • Results from one agent's work are summarized before being passed to another
Organizational context architecture and knowledge management.
  • Context systems are designed at organizational scale across teams
  • CLAUDE.md templates and conventions are standardized across projects
  • Knowledge from AI-assisted work is captured and reusable
  • Context quality is measured and improved systematically
  • Organizational learning from AI interactions is preserved beyond individual sessions
Environment Design
No configuration — uses defaults.
  • Uses Claude Code with factory default settings
  • No CLAUDE.md, no hooks, no permission customization
  • Every session starts from scratch with no project awareness
  • Approves every tool permission prompt manually
  • Does not know configuration options exist
Minimal configuration — beginning to customize.
  • May have created a basic CLAUDE.md with build commands
  • Has explored .claude/settings.json but not configured it deeply
  • Understands that configuration exists but uses it inconsistently
  • Relies on per-session context rather than persistent environment
  • Beginning to version-control AI tool configuration
Comprehensive environment with CLAUDE.md, hooks, MCP, and permissions.
  • Layered CLAUDE.md files provide context at project and directory levels
  • Hooks enforce linting, testing, and formatting on every change
  • MCP servers connect to databases, APIs, or documentation as needed
  • Permission allowlists let common operations run without interruption
  • A new team member is productive with Claude Code immediately using the configuration
Workflow-ready environment supporting automation and multi-agent operation.
  • Environment supports headless and non-interactive operation
  • CI/CD pipelines have Claude Code steps configured and tested
  • Worktree scripts and agent orchestration tools are in place
  • Token budgets and cost controls are configured for automated workflows
  • Environment configuration is tested and documented for reliability
Organization-wide environment standards and governance integration.
  • Template environments provide consistent setup across all team projects
  • Governance policies are embedded in environment configuration (deny rules, audit hooks)
  • Environment standards are versioned, reviewed, and updated organizationally
  • New projects inherit a compliant baseline configuration automatically
  • Environment metrics track adoption, compliance, and productivity impact
Workflow Scaling
Single task, single session, one prompt at a time.
  • Works on one task per session with no parallelism
  • Does not plan sequences of prompts in advance
  • Each interaction is independent with no workflow continuity
  • Cannot estimate how long a multi-step task will take with AI
  • Does not use scripts or automation for repetitive tasks
Multi-step sessions with planned prompt sequences.
  • Plans 3-5 sequential prompts before starting a complex task
  • Tracks progress through a task within a single session
  • Uses follow-up prompts to iterate and refine previous output
  • Still limited to one agent at a time in one session
  • Begins to recognize tasks that could benefit from automation
Automated single-agent tasks with scripting and headless mode.
  • Uses claude -p for scripted, non-interactive operations
  • Writes bash scripts that run Claude Code on multiple files
  • Automates repetitive tasks like documentation updates or code formatting
  • Designs multi-session workflows where each session builds on prior artifacts
  • Begins to explore running multiple agents in parallel
Multi-agent parallel workflows integrated into development pipelines.
  • Runs multiple Claude Code agents simultaneously on isolated worktrees
  • CI/CD pipelines include automated Claude Code steps for review or generation
  • Task decomposition maximizes parallel execution across agents
  • Error handling and retry logic handle agent failures gracefully
  • Cost and token usage are monitored across all automated workflows
Organizational AI workflow strategy with cross-team orchestration.
  • Workflow patterns are standardized and shared across engineering teams
  • Capacity planning accounts for AI agent compute and token budgets
  • Cross-team workflow orchestration handles complex multi-project tasks
  • Workflow effectiveness is measured with productivity and quality metrics
  • New workflow patterns are evaluated, piloted, and rolled out systematically
Safety and Governance
No safety awareness — accepts code without security review.
  • Does not review AI-generated code for security vulnerabilities
  • No audit trail for what AI changes were made and when
  • Accepts hardcoded credentials or insecure patterns without noticing
  • Does not consider the risk of AI-generated code in production
  • Cannot articulate what could go wrong with unreviewed AI output
Basic awareness — reviews for obvious security issues.
  • Checks for hardcoded secrets and credentials in diffs
  • Understands that AI-generated code can have subtle security flaws
  • Reviews authentication and authorization code more carefully
  • Knows that AI output should not be trusted for security-critical paths
  • Begins to think about what happens if AI-generated code reaches production
Project-level safety with configured permissions and automated checks.
  • Permission denylists prevent dangerous operations (rm -rf, git push --force)
  • Hooks run security scanning on every AI-generated change
  • Sensitive files (.env, credentials) are protected from AI modification
  • MCP server access is scoped to least-privilege for the task
  • Safety configuration is version-controlled and reviewed
Workflow-level safety with boundaries, monitoring, and escalation.
  • Automated agents operate within defined safety boundaries
  • Output quality monitoring detects drift before it reaches production
  • Escalation paths exist for when automated agents encounter edge cases
  • All automated agent actions are logged with full provenance
  • Safety boundaries are tested regularly to verify they actually prevent harm
Organizational governance with policies, compliance, and audit frameworks.
  • Documented policies define what AI agents can and cannot do per team/project
  • Compliance verification is integrated into deployment and release workflows
  • Audit frameworks track all AI-generated code changes with retention policies
  • Incident response procedures specifically address AI-generated code failures
  • Governance is regularly reviewed and adapted as tool capabilities evolve
