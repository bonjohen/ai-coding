Evaluation Dimensions

Cross-level dimensions used to evaluate competence across the framework.

Query Construction

The ability to write correct, readable SQL that accurately expresses data retrieval and manipulation intent.

Why It Matters

Query Construction is the foundational skill. All other dimensions build on the ability to express intent precisely in SQL.

Assessment Questions

  • Do you use CTEs rather than deeply nested subqueries?
  • Can you write a correct query when NULLs are present in join columns?
  • Do you choose JOIN types deliberately based on the data relationship?
Assess this dimension →

Data Modeling

Skill in designing relational schemas: normalization, entity relationships, constraints, and structural integrity.

Why It Matters

Schema design decisions are expensive to reverse. A poorly modeled schema constrains every query and integration built on top of it.

Assessment Questions

  • Can you identify and resolve a second normal form violation?
  • Do you add foreign key constraints as a default practice?
  • Can you explain when denormalization is the right choice?
Assess this dimension →

Analytical Techniques

Proficiency with aggregation patterns, window functions, CTEs, pivoting, and advanced data analysis constructs.

Why It Matters

Analytical Techniques determines the range of questions a practitioner can answer directly in SQL without resorting to application code.

Assessment Questions

  • Can you write a rolling 7-day average using a window function?
  • Can you compute a retention cohort table in SQL?
  • Do you know when to use FILTER (WHERE ...) vs. a CASE expression inside an aggregate?
Assess this dimension →

Performance & Optimization

Understanding of index design, execution plan analysis, query tuning, and database statistics.

Why It Matters

Queries that work correctly at 1,000 rows may fail at 10,000,000. Performance separates practitioners who can work on production systems from those who cannot.

Assessment Questions

  • Can you read an execution plan and identify why a query performs a sequential scan?
  • Do you understand how cardinality estimates affect join order selection?
  • Can you design a composite index for a specific query's filter and sort requirements?
Assess this dimension →

Data Integrity & Transactions

Command of constraints, ACID properties, isolation levels, and transactional design.

Why It Matters

A database that produces fast but inconsistent results is worse than a slow one. Data Integrity determines whether a system can be trusted under concurrent load.

Assessment Questions

  • Can you explain the difference between READ COMMITTED and REPEATABLE READ with a concrete scenario?
  • Do you understand the locking implications of adding a foreign key constraint on a large table?
  • Can you identify when optimistic locking is preferable to pessimistic locking?
Assess this dimension →

Tooling & Workflow

Use of SQL clients, schema migration tools, version control for database changes, automated testing, and CI/CD integration.

Why It Matters

SQL written outside of a version-controlled, reviewable workflow is a liability. Tooling & Workflow makes database changes auditable and safe at team scale.

Assessment Questions

  • Do all schema changes go through a migration tool?
  • Are your migrations tested in CI before reaching production?
  • Do you use a SQL linter or formatter as part of your review process?
Assess this dimension →

Collaboration & Standards

Adherence to naming conventions, schema documentation, peer review of database changes, and cross-team data contracts.

Why It Matters

Schemas outlive the teams that build them. Without consistent standards, database complexity compounds until it becomes a systemic organizational constraint.

Assessment Questions

  • Do you follow a consistent naming convention and apply it to every object?
  • Do schema changes go through peer review before production?
  • Can you describe what a data contract is and when you would use one?
Assess this dimension →
