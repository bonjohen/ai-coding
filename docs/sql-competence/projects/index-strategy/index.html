Level 4

Designing an Index Strategy

Design a complete index strategy for a workload of 5 common queries.

Scenario

An e-commerce platform has 5 frequently-run queries against its orders schema. You need to design an index strategy that covers all of them without over-indexing.

Analyze the Query Workload

List the 5 queries and identify which columns they filter, join, and sort on. Look for overlapping patterns.

A good index strategy shares indexes across queries where possible.

-- Query 1: Recent orders by customer
--   WHERE customer_id = ? AND created_at > ?
--   ORDER BY created_at DESC

-- Query 2: Order status lookup
--   WHERE customer_id = ? AND status = ?

-- Query 3: Daily revenue report
--   WHERE created_at BETWEEN ? AND ?
--   GROUP BY DATE(created_at)

-- Query 4: Fulfillment queue
--   WHERE status = 'pending' ORDER BY created_at

-- Query 5: Customer order history
--   WHERE customer_id = ? ORDER BY created_at DESC

Design the Index Set

Queries 1, 2, and 5 all filter on customer_id. A composite index (customer_id, created_at DESC) covers queries 1 and 5. Query 2 needs (customer_id, status).

Query 3 filters on created_at alone. Query 4 filters on status + created_at. Design each index with column order matching the query pattern.

Document trade-offs: each index adds write overhead. Verify no redundancy.

-- Index strategy:
CREATE INDEX idx_orders_customer_date
  ON orders (customer_id, created_at DESC);
  -- Covers: Query 1, Query 5

CREATE INDEX idx_orders_customer_status
  ON orders (customer_id, status);
  -- Covers: Query 2

CREATE INDEX idx_orders_created
  ON orders (created_at);
  -- Covers: Query 3

CREATE INDEX idx_orders_status_date
  ON orders (status, created_at)
  WHERE status = 'pending';
  -- Covers: Query 4 (partial index)

-- Trade-off: 4 indexes on one table.
-- Write overhead is acceptable because orders are
-- written once but queried many times.
