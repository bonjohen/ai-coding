Level 5

Multi-Service Data Architecture

Design data architecture for a system migrating from a monolithic database to microservices.

Scenario

Your company is splitting a monolithic application into 4 services: Users, Orders, Inventory, and Payments. All currently share one PostgreSQL database. Design the target data architecture.

Define Data Ownership Boundaries

Each service owns its data. No service directly queries another service's tables. Identify which tables belong to which service.

Shared reference data (e.g., country codes) needs a clear ownership strategy.

-- Data ownership:
-- Users Service:   users, user_addresses, user_preferences
-- Orders Service:  orders, order_items, order_status_history
-- Inventory:       products, stock_levels, warehouses
-- Payments:        payments, refunds, payment_methods

-- Shared reference: countries, currencies
-- Decision: owned by Users service, replicated read-only to others

Design the Consistency Strategy

Cross-service operations (e.g., placing an order checks inventory and creates a payment) need a consistency approach.

Decision: use the Saga pattern with compensating transactions rather than distributed 2PC. Document the trade-offs.

-- Order placement saga:
-- 1. Orders service: create order (status = 'pending')
-- 2. Inventory service: reserve stock
--    - If fails: Orders cancels order (compensating tx)
-- 3. Payments service: process payment
--    - If fails: Inventory releases stock, Orders cancels
-- 4. Orders service: confirm order (status = 'confirmed')

-- Trade-off: eventual consistency between services.
-- An order may be 'pending' for seconds during the saga.
-- Acceptable for this use case; payment-critical paths
-- have explicit compensation logic.

Plan the Migration Path

Migration from monolith to microservices happens in phases. Start by extracting the service with the fewest cross-dependencies.

Use the Strangler Fig pattern: new services read from the monolith initially, then gradually take over writes.

-- Migration phases:
-- Phase 1: Extract Users service
--   - New users DB with same schema
--   - Dual-write during transition
--   - Verify data consistency
--   - Cut over reads, then writes
--
-- Phase 2: Extract Inventory
--   - Products move to inventory DB
--   - Orders service uses API instead of JOIN
--
-- Phase 3: Extract Payments
-- Phase 4: Orders becomes standalone
--
-- Each phase: 2-4 weeks with rollback capability
