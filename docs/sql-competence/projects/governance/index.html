Level 5

Establishing Data Governance

Create a data governance framework including naming conventions, review process, and quality metrics.

Scenario

Your 30-person engineering team has inconsistent database practices: some use snake_case, others camelCase; migrations aren't always reviewed; there's no documentation standard. You need to establish governance.

Define Naming Conventions

Establish clear, enforceable naming rules for all database objects. Document with examples.

The convention should be opinionated but practical — the goal is consistency, not perfection.

-- Naming Convention Standard
--
-- Tables: snake_case, plural (users, order_items)
-- Columns: snake_case, singular (first_name, created_at)
-- Primary keys: id (serial or UUID)
-- Foreign keys: {referenced_table_singular}_id (user_id)
-- Indexes: idx_{table}_{columns} (idx_orders_customer_date)
-- Constraints: chk_{table}_{rule} (chk_bookings_end_after_start)
-- Booleans: is_{adjective} or has_{noun} (is_active, has_verified_email)
-- Timestamps: {verb}_at (created_at, updated_at, deleted_at)

Design the Review Process

Every schema change needs review. Define who reviews, what they check, and what blocks a merge.

Automate what you can: linters catch naming violations, CI verifies migrations run forward and backward.

-- Schema Change Review Checklist:
-- [ ] Migration runs forward and backward without errors
-- [ ] Naming follows convention (enforced by linter)
-- [ ] All new columns have NOT NULL or explicit DEFAULT
-- [ ] Foreign keys include ON DELETE behavior
-- [ ] New indexes justified by query workload
-- [ ] Backward-compatible (old code still works)
-- [ ] Data backfill plan documented if needed
-- [ ] Estimated lock time for ALTER on large tables
--
-- Approval: 1 reviewer for additive changes
--           2 reviewers for destructive changes (DROP, ALTER type)
