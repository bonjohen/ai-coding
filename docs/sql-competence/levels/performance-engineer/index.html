Level 4

Performance Engineer

Designs index strategies, interprets execution plans, optimizes slow queries, and manages concurrency and isolation.

Definition

The Performance Engineer treats the database as a system to be tuned, not just queried. They understand statistics, cardinality, and join algorithms.

They design composite indexes deliberately, recognize implicit type-cast index suppression, and manage concurrency through isolation level selection.

Core Concepts

Index Strategy

Designing composite and covering indexes based on query workload analysis.

Query Optimizer

Understanding how the optimizer selects plans based on statistics and cardinality estimates.

Isolation Levels

Deep understanding of READ UNCOMMITTED through SERIALIZABLE and their concurrency implications.

Partitioning

Table partitioning strategies for large datasets: range, list, and hash partitioning.

Concurrency Control

Understanding MVCC, locking, deadlock detection, and lock granularity.

Core Skills

Index Design

Can design optimal indexes for a given workload.

Observable evidence: Analyzes slow query log and creates composite indexes that eliminate table scans.

Plan Optimization

Can rewrite queries to produce better execution plans.

Observable evidence: Rewrites a correlated subquery as a join when plan shows nested loops at scale.

Concurrency Management

Selects appropriate isolation levels and locking strategies.

Observable evidence: Chooses isolation level based on consistency requirements vs throughput trade-offs.

Partition Design

Can partition large tables with appropriate partition keys.

Observable evidence: Designs time-based partitioning for a high-volume event table.

Performance Diagnosis

Can diagnose slow queries systematically.

Observable evidence: Uses execution plans, statistics, and slow query logs together.

Observable Behaviors

  • Reviews auto-generated indexes critically and removes unused ones.
  • Rewrites correlated subqueries as joins when the plan shows nested loops at scale.
  • Uses query hints only as a last resort with documented justification.
  • Understands MVCC mechanics and their impact on vacuum and replication lag.

Typical Failure Modes

  • Over-indexes tables without measuring write impact.
  • Applies query hints as a first resort instead of fixing root causes.
  • Chooses SERIALIZABLE isolation level universally without considering trade-offs.
  • Optimizes queries without understanding the full workload pattern.

Graduation Criteria

  • Designs a multi-tenant or partitioned architecture with horizontal scaling reasoning.
  • Defines an HA/DR strategy with RPO/RTO targets and replication configuration.
  • Authors SQL standards and review guidelines for a development team.
  • Evaluates SQL vs NoSQL trade-offs for a given workload and documents the decision.

Evaluation Tasks

Diagnose and Fix a Slow Query

Given a slow query execution plan and table statistics, identify the performance bottleneck, propose an optimization strategy, and validate the improvement.

Success Criteria

  • Correctly identifies the bottleneck from the execution plan.
  • Proposed index or rewrite addresses the root cause.
  • Validates improvement with before/after plan comparison.
