Level 3

Schema Designer

Designs normalized relational schemas, manages data integrity through constraints and transactions, and reads execution plans.

Definition

The Schema Designer owns the structural layer of a database. They design tables from requirements rather than retrofitting queries onto ad-hoc schemas.

They enforce integrity via constraints, understand indexing write implications, and can diagnose common performance problems at the plan level.

Core Concepts

Normalization

Understanding 1NF through 3NF and when to deviate with documented reasoning.

Constraints

Using NOT NULL, UNIQUE, CHECK, and FOREIGN KEY constraints to enforce data integrity.

Transactions

Understanding ACID properties and using explicit transaction blocks with COMMIT/ROLLBACK.

Execution Plans

Reading EXPLAIN output to understand how the database executes a query.

Stored Procedures

Writing reusable server-side logic as stored procedures, functions, and views.

Core Skills

Schema Design

Can design a normalized schema from business requirements.

Observable evidence: Produces an ER diagram and DDL from a feature specification.

Migration Writing

Can write reversible migration scripts.

Observable evidence: Writes up/down migration scripts that can be rolled back safely.

Constraint Application

Applies constraints systematically to enforce data integrity.

Observable evidence: Every new table includes appropriate NOT NULL, FK, and CHECK constraints.

Execution Plan Reading

Can read EXPLAIN output and identify seq scans vs index scans.

Observable evidence: Identifies a missing index from a sequential scan in the plan.

Transaction Management

Uses explicit transactions for multi-statement operations.

Observable evidence: Wraps related DML statements in BEGIN/COMMIT blocks.

Observable Behaviors

  • Normalizes to 3NF by default; deviates intentionally with documented reasoning.
  • Writes migration scripts that are reversible.
  • Adds foreign key constraints and understands their locking implications.
  • Reads EXPLAIN output and identifies seq scans vs index scans.

Typical Failure Modes

  • Over-normalizes schemas that need read performance.
  • Writes migrations without considering rollback.
  • Adds indexes without understanding write overhead.
  • Uses stored procedures where application logic would be more maintainable.

Graduation Criteria

  • Designs and validates an index strategy for a given query workload.
  • Demonstrates understanding of all four isolation levels with concrete failure scenarios.
  • Diagnoses and fixes a slow query using execution plan analysis.
  • Writes a partitioned table definition with appropriate partition key reasoning.

Evaluation Tasks

Design a Schema from Requirements

Given a feature specification for an e-commerce returns system, design a normalized schema with appropriate constraints and write migration DDL.

Success Criteria

  • Schema is in 3NF or has documented reasons for denormalization.
  • All foreign keys and constraints are present.
  • Migration includes both up and down scripts.
