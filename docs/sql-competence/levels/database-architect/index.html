Level 5

Database Architect

Designs enterprise data architectures, establishes governance and standards, and aligns data strategy across teams.

Definition

The Database Architect operates at the organizational level. They design for scale, resilience, and team autonomy simultaneously.

They define data contracts between services, govern naming conventions and review processes, and make buy-vs-build decisions for data infrastructure.

Core Concepts

Distributed SQL

Understanding distributed database architectures, sharding strategies, and consistency trade-offs.

Data Governance

Establishing schema review processes, naming conventions, and approval gates.

Replication & HA

Designing high-availability architectures with replication, failover, and disaster recovery.

Data Contracts

Defining cross-service data contracts that decouple teams while maintaining data integrity.

Capacity Planning

Forecasting storage, throughput, and compute requirements for database infrastructure.

Core Skills

Architecture Design

Can design data architectures that scale across teams and services.

Observable evidence: Produces architecture documents with clear trade-off analysis.

Governance Establishment

Can establish and enforce data governance practices.

Observable evidence: Implements schema review processes with clear approval criteria.

HA/DR Design

Can design high-availability and disaster recovery strategies.

Observable evidence: Defines RPO/RTO targets and tests failover procedures.

Cross-Team Alignment

Can align multiple teams on shared data models and contracts.

Observable evidence: Leads data model review sessions across service boundaries.

Technology Evaluation

Can evaluate database technologies against requirements.

Observable evidence: Produces decision documents comparing SQL and NoSQL options for a given workload.

Observable Behaviors

  • Establishes schema review processes with clear approval gates.
  • Designs data models that are stable under changing access patterns.
  • Treats backward compatibility of schema changes as a first-class constraint.
  • Leads cross-team alignment on shared reference data.

Typical Failure Modes

  • Designs theoretical architectures disconnected from practical team constraints.
  • Imposes governance overhead that slows teams without proportionate benefit.
  • Chooses distributed systems when a single well-tuned database would suffice.
  • Neglects backward compatibility in schema evolution.

Graduation Criteria

  • This is the terminal level — mastery is demonstrated through sustained organizational impact.
  • Data architectures designed remain stable through multiple product cycles.
  • Teams independently follow governance standards without constant enforcement.
  • Technology decisions are validated by production outcomes over time.

Evaluation Tasks

Design a Multi-Service Data Architecture

Design the data architecture for a system with 5+ services that share customer and transaction data. Address consistency, ownership, and migration strategy.

Success Criteria

  • Clear data ownership boundaries between services.
  • Consistency strategy documented with trade-off analysis.
  • Migration path from current state to target architecture.

Learning Emphasis

Study distributed systems theory.

Learn from post-mortems of data architecture failures.

Practice cross-team communication and governance design.

Related Dimensions

  • Data Modeling — Skill in designing relational schemas and entity relationships.
  • Performance & Optimization — Understanding of index design, execution plans, and query tuning.
  • Collaboration & Standards — Adherence to naming conventions, documentation, and data governance.
  • Data Integrity & Transactions — Command of constraints, ACID properties, and transactional design.
  • Tooling & Workflow — Use of SQL clients, migrations, version control, and CI/CD.
